# AGENTS.md — 리더회의 LLM Wiki 운영 스키마

이 파일은 본 레포의 **정본 운영 스키마**입니다. 모든 LLM 에이전트(Claude Code, Codex, Cursor, Pi 등)는 이 문서를 따라 작업합니다. `CLAUDE.md`는 이 파일을 가리키는 얇은 포인터일 뿐입니다.

---

## 1. 개요

본 레포는 **리더회의 회의록을 누적해 회사 전반의 진행 상태를 LLM이 관리하는 위키로 추적**하는 시스템입니다. 구현 패턴은 Andrej Karpathy의 [LLM Wiki gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)를 기반으로 합니다.

핵심 원리:
- **사람**은 큐레이션과 방향 제시 — raw 소스를 모으고, 질문을 던지고, 우선순위를 정합니다.
- **LLM**은 모든 grunt work — 요약, 교차참조, 페이지 작성·갱신, 인덱스·로그 유지, 일관성 점검을 담당합니다.
- 위키는 **persistent, compounding artifact**이며 매 질의마다 재구성되는 RAG가 아닙니다.

---

## 2. 3계층 구조

| 계층 | 디렉토리/파일 | 소유자 | 변경 규칙 |
|------|---------------|--------|-----------|
| Raw sources | `raw/` | 사람 | LLM은 **읽기 전용**. 절대 수정·삭제하지 않음. |
| Wiki | `wiki/` | LLM | LLM이 작성·갱신·유지보수. 사람은 읽고 피드백. |
| Schema | `AGENTS.md` (이 파일) | 함께 co-evolve | 새 패턴 발견 시 LLM이 제안 → 사람 승인 후 갱신 |

## 3. 파일 컨벤션

### 3.1 위키 페이지 frontmatter

모든 `wiki/` 내 마크다운 페이지는 상단에 YAML frontmatter를 둡니다.

```yaml
---
title: 페이지 제목
type: project | person | team | decision | concept | meeting-summary | ...
created: 2026-05-16
updated: 2026-05-16
sources:
  - raw/2026-05-16-리더회의.md
tags: []
---
```

- `type`은 고정 enum이 아니라 발견되는 대로 확장. 새 type 사용 시 본 섹션에도 추가.
- `sources`는 항상 `raw/` 기준 상대경로 리스트. 페이지 생성·갱신을 유발한 모든 원본을 기록.
- `updated`는 수정할 때마다 갱신.

### 3.2 페이지 간 링크

- 기본은 Obsidian 호환 wikilink: `[[페이지명]]`
- 표시 텍스트가 필요하면: `[[페이지명|표시 텍스트]]`
- vanilla 마크다운 링크(`[](path.md)`)도 허용 (단, 일관성 권장)

### 3.3 첨부 파일

- 이미지·녹음·PDF 등은 `raw/assets/`에 보관.
- 위키 페이지에서 참조 시 상대경로: `![[../raw/assets/foo.png]]` 또는 `![](../raw/assets/foo.png)`

### 3.4 파일·디렉토리 명명

- 한국어 파일명 허용. 자연스러운 한글 또는 `kebab-case` 모두 OK. 한 페이지 내에서 일관되게.
- raw 파일 권장 형식: `YYYY-MM-DD-주제.md` (예: `2026-05-16-주간-리더회의.md`)
- wiki 페이지 권장 형식: `엔티티명.md` (예: `프로젝트-알파.md`, `김아무개.md`)

## 4. 워크플로우

### 4.1 Ingest — 새 raw 소스 처리

사용자가 "이거 ingest해줘" 또는 유사 요청 시:

1. **소스 읽기**: `raw/`의 새 파일(들)을 읽고 핵심 정보 추출.
2. **사용자와 핵심 논의**: 무엇이 중요한지, 어디에 강조할지 짧게 확인. (사용자가 빠른 진행을 원하면 생략 가능)
3. **wiki 페이지 작성·갱신**:
   - 신규 엔티티/주제 → 새 페이지 생성 (frontmatter 포함)
   - 기존 엔티티/주제 → 해당 페이지 갱신 (`updated` 날짜 변경, `sources`에 새 raw 추가)
   - 한 소스가 보통 5~15개 페이지에 영향. 모두 빠짐없이 갱신.
4. **교차 참조 강화**: 새로 만들어진 페이지에서 기존 페이지로, 기존 페이지에서 새 페이지로 `[[wikilink]]` 추가.
5. **`wiki/index.md` 갱신**: 새 페이지를 적절한 카테고리 섹션에 등록.
6. **`wiki/log.md`에 항목 추가**: 새 항목 1개 (포맷은 §5.1).

권장: 한 번에 하나의 raw 소스만 ingest. 사용자 감독 유지.

### 4.2 Query — 위키 질의

사용자가 위키 기반 질문(예: "지난달 A 프로젝트 결정사항 뭐였지?") 시:

1. `wiki/index.md` 먼저 확인 → 관련 페이지 후보 식별.
2. 후보 페이지 읽기 → 답변 합성. 출처는 `[[]]` 링크로 인용.
3. **가치 있는 답변은 위키로 환원**: 비교표, 분석, 새 연결 등은 새 페이지로 저장 권장 (사용자 확인 후).
4. `wiki/log.md`에 query 항목 추가 (선택, 중요한 탐색만).

### 4.3 Lint — 위키 건강 점검

사용자가 "lint해줘" 요청 시:

다음 항목을 점검하고 보고서 작성:

- **모순**: 페이지 간 상충되는 주장.
- **노후 주장**: 새로운 raw가 이전 페이지의 단정을 뒤집은 경우.
- **고아 페이지**: 인바운드 `[[]]` 링크가 0인 페이지.
- **누락된 cross-reference**: 같은 엔티티를 언급하지만 링크가 없는 경우.
- **페이지 없는 중요 개념**: 여러 페이지에서 반복 언급되지만 자체 페이지가 없는 개념.
- **frontmatter 누락·불일치**: 필수 필드 빠진 경우.

보고서는 위키에 임시 페이지로 저장하지 말고 채팅에서 제시. 사용자와 함께 어떤 항목을 고칠지 결정 → 수정 → `log.md`에 lint 항목 추가.

## 5. 특수 파일

### 5.1 `wiki/log.md`

- 형식: 각 항목 헤더는 `## [YYYY-MM-DD] <op> | <title>`
  - `<op>` ∈ {`ingest`, `query`, `lint`, `schema-update`}
  - 예: `## [2026-05-16] ingest | 5월 3주차 리더회의`
- 헤더 아래 본문에는: 처리 요약, 영향받은 페이지 목록(`[[]]` 링크), 메모.
- **append-only**. 과거 항목 수정 금지.
- 빠른 조회: `grep "^## \[" wiki/log.md | tail -10`

### 5.2 `wiki/index.md`

- 카테고리별 섹션.
- 각 항목: `- [[페이지]] — 한 줄 요약`
- 초기 카테고리는 placeholder. ingest 누적되며 의미 있는 카테고리(예: `## 프로젝트`, `## 인물`, `## 분기 OKR`)로 LLM이 재편.
- 카테고리 추가·재편 시 본 AGENTS.md §6에도 반영 검토.

## 6. 스키마 진화

이 AGENTS.md는 살아있는 문서입니다.

- LLM이 ingest 중 반복 패턴(예: "결정사항이 매주 등장하므로 별도 디렉토리가 필요해 보임")을 발견하면:
  1. 채팅에서 사용자에게 변경 제안 (구체적 디렉토리 구조, 새 type, 새 카테고리 등).
  2. 승인되면 AGENTS.md 갱신 + 필요한 마이그레이션 수행.
  3. 변경 사항은 commit message에 명시: `schema: <변경 요약>`.
- 임의 변경 금지. 항상 사용자 승인 후.

## 7. 도메인 노트 (리더회의)

리더회의 특성상 자주 등장할 패턴(누적되며 추가):

- 반복 안건 (주간/격주/월간 정례 주제)
- 분기 OKR / KPI 사이클
- 결정사항(decision) 추적 — 결정 → 실행 → 결과
- 액션 아이템 follow-up
- 인적 변동(승진, 이동, 입사·퇴사)
- 조직 구조 변경

민감 정보 주의:
- 본 레포는 git history를 남깁니다. 외부에 노출되면 안 되는 정보(개인정보, 재무 세부, 보안 사고 등)는 처리 전에 사용자와 확인.
- 필요 시 `raw/private/` 같은 .gitignore 경로 도입을 사용자에게 제안.
